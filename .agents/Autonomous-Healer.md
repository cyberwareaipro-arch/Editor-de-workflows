# Autonomous Healer Skill

Use this skill to automatically fix known, tested bugs. Working in a tight loop with testers, it reads structured incident reports to apply surgical fixes to source code, preventing the Orchestrator from getting stuck on minor coding errors.

## When to use
- A formal Incident Report is generated by the Test Incident Reporter.
- A linting error causes build processes to fail.
- Syntax errors or typos are blocking execution.

## How to use

### Phase 1: Diagnosis (Analysis)
- **Consume Report:** Read the exact expected vs actual behavior from the Incident Report.
- **Locate Context:** View the specific file and line numbers referenced in the failure.

### Phase 2: Patch Application (Execution)
- **Propose Fix:** Determine if the error is a simple typo, a missing import, or a deep logical flaw. 
- **Apply Code Change:** Use surgical file-edit tools (like `sed`, `awk`, or programmatic AST manipulation) or simply rewrite the targeted function to fix the bug without breaking surrounding logic.

### Phase 3: The Healing Loop (Validation)
- **Retest:** Command the Unit Integrator Tester to re-run the specific failing test suite.
- **Evaluate:** If the test passes (exit code 0), the loop ends successfully. If it fails again, generate a new Incident Report and retry.

## Contingencies & Edge Cases
- **Loop Limit:** If the Autonomous Healer attempts to fix the same file 3 times and tests still fail, it MUST HALT to prevent hallucination loops. It should revert changes to the baseline and request human intervention.
- **Test Invalidation:** If the Healer realizes the *code* is correct according to the SRS, but the *test* was written poorly, it must fix the test script rather than altering valid business logic.

## Specifications
- **Standard:** Embodies Continuous Integration (CI) and Auto-Remediation best practices.
- **Format:** Modifies existing source (`.py`, `.ts`, etc.) and test files in real-time.
